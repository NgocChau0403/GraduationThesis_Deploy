# Task Availability Log Generation

## 1. Purpose

The two JSON logs used in the Task Availability Evaluation are generated
automatically by the backend whenever the Task Availability Validator API is
called. They are neither manually written nor generated by an LLM.

Each log provides a runtime snapshot containing:

- the dataset and class being evaluated;
- the capabilities detected from the actual PostgreSQL data;
- the availability result for each of the 52 public tasks;
- the status and failure reason of each task;
- an aggregated summary of the validation run.

The evaluation scope is:

```text
52 tasks × 2 datasets = 104 task–dataset cases
```

## 2. Source of the Logs

The task results recorded in the logs are produced by:

[`capabilityValidator.service.js`](../../../Backend/src/services/capabilityValidator.service.js)

## 4. Why `/api/tasks/available` Is Used

This endpoint:

1. finds the imported dataset from `datasetId`;
2. executes the Task Availability Validator;
3. excludes tasks that are not public or are deprecated;
4. returns the public tasks with their availability information;
5. automatically writes the validation evidence to a JSON log.

After the four thesis tasks were promoted from `experimental` to `validated`,
the endpoint returned:

```text
52 tasks for UCI
52 tasks for OULAD
```

## 5. Four-Layer Validation Architecture

Each task is evaluated through four validation layers.

### 5.1. Layer A — Structural Validation

This layer checks whether the required canonical database tables exist, such
as:

```text
student
course
class
enrollment
assessment
assessment_result
event
engagement
```

If a required table is missing:

```text
status = unsupported
```

### 5.2. Layer B — Semantic and Capability Validation

The validator reads the task requirements from `taskRegistry.json`.

Example:

```json
{
  "requiredCapabilities": [
    "assessment_scores",
    "engagement_tracking"
  ]
}
```

These requirements are compared with the capability snapshot derived from
the actual dataset.

Example for UCI:

```text
assessment_scores = true
engagement_tracking = false
```

This means that UCI contains assessment scores but does not contain positive
online learning engagement activity.

### 5.3. Layer C — Analytical Validation

This layer checks whether the data is analytically suitable for the requested
analysis, for example:

- at least two temporal points for a trend;
- at least two students for a comparison;
- sufficient variables for relationship analysis;
- required output fields for the configured visualization.

Layer C currently produces warnings and advisories. It does not directly
block task execution.

Temporal checks are scoped to `class_id` when a class is supplied, preventing
other classes in the same batch from affecting the warning.

### 5.4. Layer D — Data Sufficiency Validation

This layer queries the actual imported data and examines:

- enrollment count;
- student count;
- assessment count;
- assessment-result count;
- temporal coverage;
- positive engagement activity.

Example for UCI:

```text
students = 649
assessment results = 1947
positive engagement rows = 0
```

Therefore, score-based tasks may be executable while tasks that require
engagement evidence are unavailable because of insufficient runtime data.

## 6. Special Runtime Check for Engagement

`engagement_tracking` is checked in both Layer B and Layer D because the
existence of the canonical `engagement` table does not prove that meaningful
activity exists.

An imported dataset may contain the table while every `engagement_count`
value is zero.

The responsibilities are therefore separated as follows:

- Layer B checks whether the capability is semantically present.
- Layer D requires at least one row with `engagement_count > 0`.
- If engagement is required but no positive activity exists, the task receives
  `status = insufficient_data`.

Other capabilities, including `registration_timing`, `absence_tracking`, and
`socioeconomic_context`, are currently presence-based semantic checks. They
should only be promoted to Layer D checks when explicit task-specific
sample-size requirements have been defined.

## 7. Exact Capability Rules

The relevant capabilities are computed using the following runtime rules:

```text
engagement_tracking =
positive_engagement_rows >= 1

temporal_activity =
positive_engagement_rows >= 1
AND distinct_engagement_weeks >= 2

resource_clickstream =
positive_engagement_rows >= 1
AND resource_type_non_null_rows >= 1

disadvantage_scoring =
disadvantage_score_students >= 2
```

The `disadvantage_scoring >= 2` threshold is used because the capability
supports comparison or cohort-level analysis. A single scored student is not
sufficient to provide comparative evidence.

## 8. Status Derivation

The validator currently derives the final status using this precedence:

```javascript
if (structural === "fail") return "unsupported";
if (data_sufficiency === "fail") return "insufficient_data";
if (semantic === "fail") return "partial";
return "executable";
```

| Condition | Current status | Meaning |
|---|---|---|
| Structural failure | `unsupported` | A required canonical table or structure is missing. |
| Data-sufficiency failure | `insufficient_data` | The structure exists, but runtime evidence is insufficient. |
| Semantic failure | `partial` | One or more required capabilities are unavailable or incompatible. |
| No layer fails | `executable` | The task is allowed to execute. |

The `firstFailedLayer` order used in the log follows the same precedence:

```text
structural → data_sufficiency → semantic → analytical
```

## 9. Interpretation of `partial`

The current name `partial` can be misleading. A task with this status is not
clickable because the system only enables tasks whose status is:

```text
executable
```

In the current implementation, `partial` may mean that a required semantic
capability is missing entirely. It does not necessarily mean that a reduced
version of the analysis can be executed.

For clearer future semantics, the recommended status taxonomy is:

```text
unsupported       = required database structure is missing
insufficient_data = structure exists, but runtime data is insufficient
incompatible      = required capability or dataset compatibility is missing
partial           = a defined fallback can produce reduced output
executable        = the complete task can run
```

Changing `partial` to `incompatible` would affect existing logs, UI labels,
ground-truth labels, and evaluation calculations. For the current thesis
evaluation, the implementation keeps `partial`, but it must be documented as
a non-executable semantic incompatibility rather than guaranteed partial
execution.

## 10. Corrected A-B04 Availability Contract

Task `A-B04` is the cohort-level at-risk overview. Its query returns:

```text

```

Its corrected availability contract is:

```json
{
  "required_all": [
    "assessment_scores",
    "engagement_tracking"
  ],
  "chart_required_fields": [
    "at_risk_label",
    "student_count"
  ]
}
```

The previous contract incorrectly contained OULAD weekly-engagement fields
and was removed because it did not match the task query or visualization.

## 12. Log Storage

The log service writes the files to:

```text
Docs/evaluation_logs/task_availability/
```

The filename contains:

```text
task_availability
+ timestamp
+ random identifier
```

## 13. Evaluation Role

The logs provide runtime evidence that can be used to:

- demonstrate that task availability is checked automatically;
- explain why a task is enabled or blocked;
- compare machine results with independently defined ground truth;
- construct a confusion matrix;
- calculate Accuracy, Precision, Recall, and F1-score;
- preserve reproducible evidence for the thesis appendix.

The logs demonstrate consistency between the task registry, capability
snapshot, validation layers, and machine-generated status. However, validator
accuracy should only be claimed when the ground truth was defined
independently from the same runtime output.

## 14. Conclusion

The Task Availability log is an automatically generated backend artifact. It
records the result of comparing:

```text
Task requirements in taskRegistry.json
                    +
Capabilities and actual PostgreSQL data
                    ↓
Availability status for each task
```

The corrected implementation keeps structural, semantic, analytical, and
data-sufficiency concerns separate, aligns the displayed failed layer with
status precedence, scopes temporal checks to the selected class, and records
the exact capability thresholds used during validation.
